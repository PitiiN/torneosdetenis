# Handling Supabase "Never" Type Inference

In Next.js API routes or Server Components, the Supabase client may sometimes infer query results as `never[]` or specific fields as `never`, particularly when use `select()` with a string argument that the compiler cannot statically link to the database schema.

## Problem
You receive errors like:
`Property 'start_at' does not exist on type 'never'.`

This happens even if the database schema is correctly defined, because the dynamic nature of the `select()` query breaks the static type inference chain.

## Solution: Explicit Type Assertions

Define an interface for the expected record shape and use a type assertion on the `data` result.

### Example Implementation

```typescript
// 1. Define the record interface
interface BookingRecord {
    id: string
    start_at: string
    end_at: string
    status: string
}

export async function GET(request: NextRequest) {
    const supabase = await createClient()
    
    // 2. Perform the query
    const { data: bookingsData, error } = await supabase
        .from('bookings')
        .select('id, start_at, end_at, status')
        .eq('field_id', fieldId)

    // 3. Cast to the expected record type
    const bookings = bookingsData as BookingRecord[] | null

    // 4. Safely access properties
    bookings?.forEach((booking) => {
        const start = booking.start_at // No more 'never' error
    })
}
```

## Why this is preferred
While Supabase can generate types from the database, manually defining small interfaces for API results provides:
1. **Decoupling**: The API response shape is explicitly controlled.
2. **Reliability**: Fixes intermittent linting errors where the compiler loses track of the schema.
3. **Simplicity**: Avoids complex generic chains for simple data fetching.
